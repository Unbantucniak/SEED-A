# 经验单元与元属性体系设计
## 第一阶段核心交付物

### 1. 经验单元核心结构定义
经验单元是对智能体全开发交互过程的结构化抽象，包含以下核心字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| experience_id | string | 全局唯一经验ID（UUID格式） |
| task_intent | object | 任务意图信息：包含原始需求、用户指令、任务类型（需求澄清/代码生成/调试修复/评审优化等） |
| context_state | object | 上下文状态：包含当前代码仓库快照、依赖版本、环境配置、历史交互上下文 |
| operation_sequence | array | 操作序列：智能体执行的所有动作（工具调用、代码修改、指令输出等），每个动作包含动作类型、输入参数、输出结果 |
| constraints | array | 环境约束：任务执行过程中的限制条件（编程语言、框架版本、性能要求、安全规范等） |
| execution_result | object | 执行结果：包含最终输出、是否成功、错误信息、用户反馈、执行耗时、资源开销 |
| created_at | timestamp | 经验生成时间 |
| last_used_at | timestamp | 最后一次被调用时间 |

### 2. 多维元属性评价体系
为每个经验单元附加多维度元属性，用于路由决策与演化管理：

#### 2.1 静态元属性（生成时确定，不随使用变更）
| 属性名 | 类型 | 取值范围 | 说明 |
|--------|------|----------|------|
| source_credibility | float | 0-1 | 来源可信度：1=用户确认正确的经验，0=完全未验证的自动生成经验 |
| domain_tag | array | 字符串数组 | 领域标签：如"Python后端开发"、"React前端"、"数据库优化"、"bug修复"等 |
| complexity | int | 1-5 | 任务复杂度：1=简单单步任务，5=复杂多阶段项目级任务 |
| generalization | float | 0-1 | 通用性：1=可在多个场景复用，0=仅适用于当前特定场景 |

#### 2.2 动态元属性（随使用过程动态更新）
| 属性名 | 类型 | 取值范围 | 说明 |
|--------|------|----------|------|
| success_rate | float | 0-1 | 成功率：复用该经验后任务成功的比例 |
| use_count | int | >=0 | 使用次数：该经验被调用的总次数 |
| average_benefit | float | >=0 | 平均收益：复用该经验相比无经验基线节省的时间/交互轮次的平均值 |
| timeliness | float | 0-1 | 时效性：1=完全符合当前技术栈版本，0=已完全过期失效（如依赖的框架版本已停止维护） |
| conflict_score | float | 0-1 | 冲突度：与其他经验存在冲突的程度，越高表示越可能与其他经验产生矛盾 |

### 3. 经验图谱结构定义
经验图谱采用有向属性图存储，节点与边定义如下：
- **节点**：每个节点对应一个经验单元，节点属性为上述经验单元字段+元属性
- **边类型**：
  1. `dependency`：前置依赖关系，经验A→经验B表示完成经验B对应的任务需要先完成经验A对应的任务
  2. `similarity`：语义相似关系，边权重为两个经验的语义相似度（0-1）
  3. `causality`：因果关联关系，边权重为经验A的使用对经验B成功率的提升幅度
  4. `derivation`：派生关系，经验A→经验B表示经验B是在经验A的基础上修改优化而来

### 4. 元属性计算规则
#### 初始赋值规则
- 新生成的经验初始`success_rate=0.5`、`use_count=0`、`timeliness=1.0`、`conflict_score=0`
- 来源为用户确认正确的经验初始`source_credibility=1.0`，自动生成未验证的初始`source_credibility=0.3`
- 通用性通过经验领域标签的覆盖范围自动计算

#### 更新规则
- 每次经验被调用并完成任务后：
  - `use_count +=1`
  - `success_rate = (success_rate * (use_count-1) + 任务成功?1:0) / use_count`
  - `last_used_at`更新为当前时间
  - 重新计算`average_benefit`
- 每7天定时更新所有经验的`timeliness`：根据依赖的技术栈版本发布时间与当前时间差计算，越旧的经验时效性越低
- 每15天定时计算经验之间的`conflict_score`：对相似经验的执行结果差异进行统计，差异越大冲突度越高
